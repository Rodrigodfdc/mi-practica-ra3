01 ——

¿Qué es GitHub Pages?

GitHub Pages es un servicio de hosting estático integrado directamente en GitHub. Permite publicar sitios web a partir de los archivos de un repositorio sin necesidad de configurar ningún servidor externo, dominio de pago ni pipeline de CI/CD complejo.

El flujo es simple: subes tu código HTML/CSS/JS a un repositorio, activas Pages en los ajustes, y GitHub se encarga del resto. En cuestión de minutos tu sitio estará accesible en https://<usuario>.github.io/<repo>.

🗂️

Basado en Git

Cada push a la rama configurada activa el despliegue automáticamente.

⚡

Gratis

Repositorios públicos pueden usar Pages sin ningún coste adicional.

🔒

HTTPS automático

GitHub gestiona el certificado TLS. Tu sitio siempre sirve en HTTPS.

🌐

CDN global

El contenido se distribuye desde los servidores de GitHub a nivel mundial.

02 ——

Pasos para publicar

01

Crear una cuenta en GitHub

Accede a github.com y regístrate con tu correo. Elige un nombre de usuario que sea el mismo que aparecerá en tu URL de Pages.

02

Crear un nuevo repositorio

Haz clic en el botón New repository. Dale un nombre descriptivo, márcalo como Public e inicialízalo con un README. Solo los repositorios públicos tienen acceso gratuito a Pages.

03

Crear el archivo index.html

El archivo de entrada de tu sitio debe llamarse index.html. Puedes crearlo directamente en GitHub con el botón Add file → Create new file o subirlo desde tu máquina.

04

Subir los archivos al repositorio

Haz commit del archivo. Si trabajas en local, puedes usar git add . && git commit -m "Add site" && git push. Los cambios se reflejan en la rama main.

05

Activar GitHub Pages

Ve a Settings → Pages en tu repositorio. En Source selecciona la rama main y la carpeta / (root). Pulsa Save.

06

Acceder a tu sitio publicado

Tras unos segundos (o minutos la primera vez), GitHub generará la URL y la mostrará en la sección Pages. Tu sitio estará en https://<usuario>.github.io/<repo>.

03 ——

Estructura del proyecto

Un sitio estático básico puede componerse de un único archivo, pero una estructura ordenada facilita el mantenimiento:

TREE 
mi-repositorio/
├── index.html          # Punto de entrada obligatorio
├── css/
│   └── styles.css      # Estilos separados (opcional)
├── js/
│   └── main.js         # Scripts (opcional)
├── img/
│   └── logo.png        # Recursos de imagen
└── README.md           # Descripción del proyecto
💡 Importante

GitHub Pages sirve el archivo index.html que esté en la raíz del repositorio (o en la carpeta /docs si así lo configuras). Sin ese archivo, el sitio no tendrá punto de entrada.

Un ejemplo mínimo de index.html:

HTML 
<!-- Estructura mínima -->
<!DOCTYPE html>
<html lang="es">
  <head>
    <meta charset="UTF-8">
    <title>Mi sitio</title>
  </head>
  <body>
    <h1>Hola mundo</h1>
  </body>
</html>

Ventajas e inconvenientes

Aspecto Descripción Valoración
Coste Totalmente gratuito para repos públicos ✓ Ventaja
Configuración Sin instalación de servidor ni DNS manual ✓ Ventaja
Integración CI/CD Despliegue automático en cada push ✓ Ventaja
Backend Solo sitios estáticos; sin PHP, Node, bases de datos △ Limitación
Repositorios privados Requieren plan de pago para usar Pages △ Limitación
Límite de tamaño Repositorio máximo de 1 GB, archivos hasta 100 MB △ Limitación
05 ——

Conceptos clave

Sitio estático vs dinámico

Un sitio estático sirve archivos HTML/CSS/JS tal cual están almacenados: sin procesamiento en servidor, sin base de datos. Un sitio dinámico genera el HTML en tiempo de ejecución según la petición, el usuario o los datos.

Rama y carpeta fuente

GitHub Pages puede servir el contenido desde la raíz de la rama main o desde una subcarpeta /docs. Esto permite mantener el código fuente y la documentación publicada en el mismo repositorio sin mezclar ficheros.

Caché y tiempo de despliegue

Tras hacer push, el despliegue suele tardar entre 30 segundos y 2 minutos. Si el sitio no refleja los cambios, fuerza la recarga con Ctrl + Shift + R para limpiar la caché del navegador.

⚙️ Tip de workflow

Trabaja en una rama de desarrollo (dev) y haz merge a main solo cuando el sitio esté listo. Así, Pages solo desplegará versiones estables.
